Quick Start Guide

Get your first model-agnostic authorization working in 5 minutes. This guide uses Article as an example, but works with any Eloquent model.

Key Concept: Unlike traditional systems where users have permissions, this package lets you attach abilities directly to any entity - Articles, Organizations, TeamMembers, etc. The entity itself controls what can be done to it.

Step 1: Install the Package

Install via Composer:

composer require akindutire/authorization-pkg

The package auto-registers via Laravel's package discovery.

Step 2: Add Trait to Your Model

Add the HasPermissions trait to any model that needs abilities:

use Akindutire\Authorization\Traits\HasPermissions;
use Illuminate\Database\Eloquent\Model;

class Article extends Model
{
    use HasPermissions;

    protected $casts = [
        'allowed_permissions' => 'array',  // Abilities this article has
        'revoked_permissions' => 'array',  // Explicitly denied abilities
    ];
}
Model-Agnostic: This works with ANY model - Article, TeamMember, Organization, User, Product, etc. Each entity manages its own abilities.

Step 3: Create Database Migration

Generate a migration to add ability columns to your table:

# For articles table
php artisan make:permission-migration articles

# For other models
php artisan make:permission-migration team_members
php artisan make:permission-migration organizations

Run the migration:

php artisan migrate

This adds two JSON columns to your table:

  • allowed_permissions - Abilities the entity has
  • revoked_permissions - Explicitly denied abilities

Step 4: Grant Abilities to Entities

Attach abilities to your entities (not users!):

// Grant abilities to an Article
$article = Article::find(1);
$article->grantPermission(['can_be_edited', 'can_be_published']);

// Grant abilities to a TeamMember
$member = TeamMember::find(1);
$member->grantPermission(['can_invite', 'can_manage_billing']);

// Grant abilities to an Organization
$org = Organization::find(1);
$org->grantPermission(['can_use_api', 'can_export_data']);
Entity-Centric: Abilities belong to the entity itself. An article with can_be_edited ability can be edited. A team member with can_invite ability can invite others.

Step 5: Protect Controller Methods

Use PHP 8 attributes to check if entities have required abilities:

use Akindutire\Authorization\Attributes\{HasAny, HasAll, SubjectValue};
use App\Models\Article;

class ArticleController extends Controller
{
    /**
     * Check if the ARTICLE has the ability to be edited
     * (NOT checking if a user can edit)
     */
    #[HasAny(['can_be_edited'], Article::class, 'id')]
    public function update(
        #[SubjectValue] int $id,
        Request $request
    ) {
        // Only executes if the Article has 'can_be_edited' ability
        $article = Article::find($id);
        $article->update($request->validated());

        return response()->json($article);
    }

    /**
     * Article must have BOTH abilities
     */
    #[HasAll(['can_be_published', 'can_be_featured'], Article::class, 'id')]
    public function publish(#[SubjectValue] int $id)
    {
        $article = Article::find($id);
        $article->update(['published_at' => now()]);

        return response()->json($article);
    }

    /**
     * Works with any property, not just 'id'
     */
    #[HasAny(['can_be_viewed'], Article::class, 'slug')]
    public function show(#[SubjectValue] string $slug)
    {
        return Article::where('slug', $slug)->firstOrFail();
    }
}

Understanding the Attributes

  • #[HasAny(['can_be_edited'])] - Entity needs at least ONE of the listed abilities
  • #[HasAll(['can_be_published', 'can_be_featured'])] - Entity needs ALL listed abilities
  • Article::class - The model class to check (Article, TeamMember, Organization, etc.)
  • 'id' - The property to lookup the entity by (can be 'id', 'uuid', 'slug', etc.)
  • #[SubjectValue] - Marks the parameter containing the lookup value

Step 6: Register Middleware

Add the validation middleware to your routes:

// app/Http/Kernel.php
protected $middlewareAliases = [
    // ... other middleware
    'validate.subject.action' => \Akindutire\Authorization\Middleware\ValidateSubjectAction::class,
];

// routes/web.php or routes/api.php
Route::middleware(['validate.subject.action'])->group(function () {
    Route::put('/articles/{id}', [ArticleController::class, 'update']);
    Route::post('/articles/{id}/publish', [ArticleController::class, 'publish']);
    Route::get('/articles/{slug}', [ArticleController::class, 'show']);
});

Step 7: Check Abilities Programmatically

For complex logic, use model methods or the facade:

// Using model methods
$article = Article::find(1);

if ($article->hasPermission('can_be_edited')) {
    // Article can be edited
}

if ($article->hasAllPermissions(['can_be_published', 'can_be_featured'])) {
    // Article can be published and featured
}

if ($article->hasAnyPermission(['can_be_edited', 'can_be_viewed'])) {
    // Article has at least one of these abilities
}

// Using the facade
use Akindutire\Authorization\Facades\EntityPermission;

$canEdit = EntityPermission::subject($article)->hasAny(['can_be_edited']);
$canPublish = EntityPermission::subject($article)->hasAll(['can_be_published']);

Complete Working Example

Here's a complete flow showing how abilities work with Articles:

// 1. Create an article
$article = Article::create([
    'title' => 'My First Article',
    'slug' => 'my-first-article',
    'content' => 'Article content...',
]);

// 2. Grant abilities to the article
$article->grantPermission(['can_be_edited', 'can_be_published']);

// 3. Protect your routes
Route::middleware(['validate.subject.action'])->group(function () {
    Route::put('/articles/{id}', [ArticleController::class, 'update']);
    Route::post('/articles/{id}/publish', [ArticleController::class, 'publish']);
    Route::get('/articles/{slug}', [ArticleController::class, 'show']);
});

// 4. Add attributes to controller methods
class ArticleController extends Controller
{
    // This checks if the ARTICLE has 'can_be_edited' ability
    #[HasAny(['can_be_edited'], Article::class, 'id')]
    public function update(#[SubjectValue] int $id, Request $request)
    {
        $article = Article::find($id);
        $article->update($request->validated());
        return response()->json($article);
    }

    // This checks if the ARTICLE has both abilities
    #[HasAll(['can_be_published'], Article::class, 'id')]
    public function publish(#[SubjectValue] int $id)
    {
        $article = Article::find($id);
        $article->update(['published_at' => now()]);
        return response()->json($article);
    }

    // This checks by slug instead of id
    #[HasAny(['can_be_viewed'], Article::class, 'slug')]
    public function show(#[SubjectValue] string $slug)
    {
        return Article::where('slug', $slug)->firstOrFail();
    }
}

// 5. Make requests
// ✅ PUT /articles/1 - Works (article has 'can_be_edited')
// ✅ POST /articles/1/publish - Works (article has 'can_be_published')
// ✅ GET /articles/my-first-article - Works if granted 'can_be_viewed'

// 6. Revoke an ability
$article->revokePermission(['can_be_edited']);

// ❌ PUT /articles/1 - Now returns 403 (article no longer has 'can_be_edited')

Works with Any Model

The same approach works with any Eloquent model:

// TeamMember abilities
$member = TeamMember::find(1);
$member->grantPermission(['can_invite', 'can_manage_billing']);

#[HasAny(['can_invite'], TeamMember::class, 'id')]
public function invite(#[SubjectValue] int $id) {
    // Only executes if TeamMember has 'can_invite' ability
}

// Organization abilities
$org = Organization::find(1);
$org->grantPermission(['can_use_api', 'can_export_data']);

#[HasAll(['can_use_api'], Organization::class, 'org_id')]
public function apiAccess(#[SubjectValue] int $org_id) {
    // Only executes if Organization has 'can_use_api' ability
}

Optional: Define Ability Templates

For reusable ability sets, define templates in config/akindutire-authorization.php:

return [
    'abilities' => [
        'article_editor' => ['can_be_edited', 'can_be_deleted'],
        'article_publisher' => ['can_be_edited', 'can_be_published', 'can_be_featured'],
        'team_admin' => ['can_invite', 'can_remove', 'can_manage_billing'],
        'org_premium' => ['can_use_api', 'can_export_data', 'can_white_label'],
    ],
];

Then use them:

use Akindutire\Authorization\Facades\EntityPermission;

// Grant template abilities
$article->grantPermission(EntityPermission::getAbilities('article_publisher'));
$member->grantPermission(EntityPermission::getAbilities('team_admin'));
$org->grantPermission(EntityPermission::getAbilities('org_premium'));
You're Done! You now have a fully functional model-agnostic authorization system. Abilities belong to entities, validation is automatic, and caching is handled for you.

Key Takeaways

  • Abilities belong to entities (Article, TeamMember, Organization), not users
  • #[HasAny] and #[HasAll] attributes check if the entity has abilities
  • Works with any Eloquent model by adding the HasPermissions trait
  • Lookup entities by any property: id, uuid, slug, email, etc.
  • Automatic caching with auto-invalidation - no manual cache clearing needed
