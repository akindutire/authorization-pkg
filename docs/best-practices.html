Best Practices

Follow these guidelines for optimal performance, security, and maintainability in production.

Model-Agnostic: All best practices apply to any Eloquent model (Article, TeamMember, Organization, etc.). Build scalable authorization for all your entities.

1. Explicitly Define $casts for Better IDE Support

✅ Do (Recommended):

use Akindutire\Authorization\Traits\HasPermissions;

class Article extends Model
{
    use HasPermissions;

    // Explicit casts provide better IDE autocomplete and type inference
    protected $casts = [
        'allowed_permissions' => 'array',
        'revoked_permissions' => 'array',
    ];
}

class TeamMember extends Model
{
    use HasPermissions;

    protected $casts = [
        'allowed_permissions' => 'array',
        'revoked_permissions' => 'array',
    ];
}

class Organization extends Model
{
    use HasPermissions;

    protected $casts = [
        'allowed_permissions' => 'array',
        'revoked_permissions' => 'array',
    ];
}

⚠️ Acceptable (but less ideal):

class Article extends Model
{
    use HasPermissions;
    // Trait auto-adds casts, but IDE won't know the types
}
The trait automatically adds these casts if not defined. Explicit definition is recommended for better IDE support and code clarity.

2. Use Entity-Specific Enums for Type Safety

✅ Do (Best for large applications):

// Define entity-specific ability enums
enum ArticleAbilities: string {
    case CAN_BE_EDITED = 'can_be_edited';
    case CAN_BE_DELETED = 'can_be_deleted';
    case CAN_BE_PUBLISHED = 'can_be_published';
    case CAN_BE_FEATURED = 'can_be_featured';
}

enum TeamMemberAbilities: string {
    case CAN_INVITE = 'can_invite';
    case CAN_REMOVE = 'can_remove';
    case CAN_MANAGE_BILLING = 'can_manage_billing';
}

enum OrganizationAbilities: string {
    case CAN_USE_API = 'can_use_api';
    case CAN_EXPORT_DATA = 'can_export_data';
    case CAN_WHITE_LABEL = 'can_white_label';
}

// Use in attributes
#[HasAny([ArticleAbilities::CAN_BE_EDITED->value], Article::class, 'id')]
public function update(#[SubjectValue] int $id) { }

#[HasAny([TeamMemberAbilities::CAN_INVITE->value], TeamMember::class, 'id')]
public function invite(#[SubjectValue] int $id) { }

#[HasAll([OrganizationAbilities::CAN_USE_API->value], Organization::class, 'org_id')]
public function apiAccess(#[SubjectValue] int $org_id) { }

Benefits:

  • IDE autocomplete for all available abilities per entity type
  • Compile-time checking prevents typos
  • Centralized ability definitions per entity type
  • Easy refactoring across entity types
  • Clear separation between different entity abilities

✅ Also acceptable (for smaller applications):

#[HasAny(['can_be_edited'], Article::class, 'id')]
#[HasAny(['can_invite'], TeamMember::class, 'id')]
#[HasAll(['can_use_api'], Organization::class, 'org_id')]

3. Configure Indexed Properties

✅ Do:

// config/akindutire-authorization.php
'indexed_properties' => ['uuid', 'slug', 'tenant_id'],

// Use them in attributes for different entities
#[HasAny(['can_be_published'], Article::class, 'slug')]
#[HasAny(['can_invite'], TeamMember::class, 'uuid')]
#[HasAny(['can_use_api'], Organization::class, 'tenant_id')]

❌ Don't:

// Using unindexed property
#[HasAny(['can_be_edited'], Article::class, 'custom_field')]
// Without adding 'custom_field' to indexed_properties
Performance Impact: Unindexed properties result in full table scans. Add all lookup properties to indexed_properties for optimal performance.

4. Use Atomic Operations for Race-Condition Safety

✅ Do (Race-condition safe):

// Article abilities - atomic operations
$article = Article::find(1);
$article->grantPermission('can_be_published');
$article->grantPermission(['can_be_edited', 'can_be_featured']);

// TeamMember abilities - atomic operations
$member = TeamMember::find(1);
$member->grantPermission(['can_invite', 'can_manage_billing']);

// Organization abilities - atomic operations
$org = Organization::find(1);
$org->grantPermission(['can_use_api', 'can_export_data']);

❌ Don't (Race condition possible):

// Manual array manipulation - NOT atomic
$perms = json_decode($article->allowed_permissions, true);
$perms[] = 'can_be_published';
$article->allowed_permissions = json_encode($perms);
$article->save(); // Race condition possible!
Why it matters: In high-concurrency scenarios (multiple admins updating abilities simultaneously), atomic operations via grantPermission() prevent lost updates.

5. Warm Cache in Production Deployments

✅ Do (Best performance):

#!/bin/bash
# deploy.sh

# Standard Laravel deployment
php artisan down
git pull origin main
composer install --no-dev --optimize-autoloader

# Run migrations
php artisan migrate --force

# Warm permission cache (eliminates first-request overhead)
php artisan permission:cache

# Optimize Laravel
php artisan config:cache
php artisan route:cache
php artisan view:cache

php artisan up

✅ Also acceptable (with auto-invalidation enabled - default):

# deploy.sh
php artisan migrate
# Auto-invalidation handles cache updates automatically
# First request per route will be slightly slower (~50μs reflection overhead)
With auto-invalidation enabled (default), cache warming is optional. It only improves first-request performance by ~50μs.

6. Use Redis for Production Caching

✅ Do (Required for production):

# .env
CACHE_DRIVER=redis
REDIS_HOST=127.0.0.1
REDIS_PORT=6379
REDIS_PASSWORD=null
REDIS_DB=0

PERMISSION_CACHE_TTL=300

❌ Don't (Not suitable for production):

# .env
CACHE_DRIVER=file  # Slow, no cross-server sharing
CACHE_DRIVER=array # Lost on every request

Why Redis is essential:

  • Multi-server deployments with shared cache
  • Sub-millisecond cache reads (<1ms)
  • 99%+ cache hit rates in production
  • Automatic cache expiration via TTL
File cache doesn't work across multiple servers, and array cache doesn't persist between requests.

7. Configure All Entity Tables and Properties

✅ Do:

// config/akindutire-authorization.php
return [
    // All tables with HasPermissions trait
    'indexed_tables' => [
        'articles',
        'team_members',
        'organizations',
        'users',
    ],

    // All properties used in attribute lookups
    'indexed_properties' => [
        'uuid',
        'slug',
        'tenant_id',
    ],

    // All properties for cache invalidation
    'cache_keys' => [
        'id',
        'uuid',
        'slug',
        'tenant_id',
    ],
];

❌ Don't:

// Incomplete configuration
'indexed_tables' => ['users'], // Missing articles, team_members, organizations
'indexed_properties' => ['uuid'], // Missing slug, tenant_id used in lookups
'cache_keys' => [], // Cache won't invalidate properly

Why it matters:

  • indexed_tables: Creates database indexes for fast queries (>100x faster)
  • indexed_properties: Indexes specific columns used in WHERE clauses
  • cache_keys: Ensures multi-key cache invalidation when abilities change

8. Enable Auto-Invalidation for Reflection Cache

✅ Do (Default configuration):

// config/akindutire-authorization.php
'auto_invalidate_reflection_cache' => true,

Benefits:

  • Automatic cache updates when attributes change
  • No manual cache clearing during development
  • Minimal overhead (~5-10μs on first request)
  • Production-safe with warmed caches

❌ Don't (Legacy behavior):

// config/akindutire-authorization.php
'auto_invalidate_reflection_cache' => false, // Requires manual cache clearing

9. Optimize Cache TTL Based on Usage Patterns

High-Traffic, Read-Heavy Applications:

# .env
PERMISSION_CACHE_TTL=3600  # 1 hour - maximize cache hits

Expected results: 99%+ cache hit rate, <1ms authorization checks

Applications with Frequent Ability Changes:

# .env
PERMISSION_CACHE_TTL=60  # 1 minute - balance freshness and performance

Expected results: 95%+ cache hit rate, ~2-3ms authorization checks

Development Environment:

# .env
CACHE_DRIVER=array
PERMISSION_CACHE_TTL=0  # No caching - always fresh data

10. Use Ability Templates for Consistency

✅ Do:

// config/akindutire-authorization.php
'abilities' => [
    // Article templates
    'article_editor' => ['can_be_edited', 'can_be_deleted'],
    'article_publisher' => ['can_be_edited', 'can_be_published', 'can_be_featured'],

    // Team member templates
    'team_admin' => ['can_invite', 'can_remove', 'can_manage_billing'],
    'team_member' => ['can_invite', 'can_view_analytics'],

    // Organization templates
    'org_premium' => ['can_use_api', 'can_export_data'],
    'org_enterprise' => ['can_use_api', 'can_export_data', 'can_white_label', 'can_sso'],
],

// Usage
use Akindutire\Authorization\Facades\EntityPermission;

$article->grantPermission(EntityPermission::getAbilities('article_publisher'));
$member->grantPermission(EntityPermission::getAbilities('team_admin'));
$org->grantPermission(EntityPermission::getAbilities('org_enterprise'));

Benefits:

  • Consistent ability sets across entity types
  • Easy upgrades (e.g., org_free → org_premium)
  • Centralized ability definitions
  • Self-documenting authorization structure

Troubleshooting

Cache Not Working

Symptoms: Ability changes not reflecting immediately, or slow ability checks

Solutions:

# 1. Verify Redis is operational
redis-cli ping  # Should return PONG

# 2. Check cache configuration
php artisan config:cache

# 3. Clear all caches
php artisan cache:clear
php artisan permission:cache-clear

# 4. Verify model has explicit $casts
# Check that your models include:
protected $casts = [
    'allowed_permissions' => 'array',
    'revoked_permissions' => 'array',
];

# 5. Check cache keys in config
# Ensure 'cache_keys' includes all lookup properties

Subject Value Not Found

Error: "Subject value not found, ensure to set the subject value on the parameter using the SubjectValue attribute"

Cause: The specified key doesn't exist in the request

Solutions:

// 1. Check extraction priority (in order):
// input() → route() → query() → json()

// 2. Verify request contains the key
#[SubjectValue('article_id')]  // Ensure 'article_id' exists in request

// 3. Check route parameters match
Route::put('/articles/{article_id}', ...);  // Must match SubjectValue key

// 4. Debug the request
dd($request->all());  // See what's actually in the request

Slow Ability Checks (>100ms)

Diagnosis:

// Enable query log to count database queries
DB::enableQueryLog();

$article->hasPermission('can_be_edited');
$queries = DB::getQueryLog();

if (count($queries) > 1) {
    // Multiple queries = caching not working properly
    dd($queries);  // Inspect what queries are running
}

Solutions:

# 1. Verify Redis is running and configured
redis-cli ping
php artisan config:clear

# 2. Warm reflection cache
php artisan permission:cache

# 3. Add database indexes for lookup properties
# Ensure all properties in indexed_properties config have database indexes

# 4. Increase cache TTL
# Update PERMISSION_CACHE_TTL in .env to 300-600 seconds

# 5. Check indexed_tables configuration
# Ensure all entity tables are listed in indexed_tables

Abilities Not Persisting

Symptoms: grantPermission() doesn't save abilities

Solutions:

// 1. Check model has $casts defined
protected $casts = [
    'allowed_permissions' => 'array',
    'revoked_permissions' => 'array',
];

// 2. Verify columns exist in database
php artisan migrate:status

// 3. Check fillable/guarded
// Ensure permission columns aren't blocked:
protected $guarded = [];  // Or add to $fillable

// 4. Test manually
$article = Article::find(1);
$article->grantPermission('can_be_edited');
dd($article->fresh()->allowed_permissions);  // Should show the ability
Production Ready: Following these best practices ensures your authorization layer adds <1ms overhead per request even under heavy load with millions of entities.
